03 What Was Built

Unified Control Panel — lab-control.py

Single Python script managing the entire lab. Dynamic VM status banner via virsh. Start ALL / Stop ALL (force-kills stragglers with virsh destroy). K3s start/stop/status. Scale up/down workers (Terraform → SSH ready → k3s join → image sync → replica scaling). Ansible playbook runner. Nuke Test — scales app to 0, waits for alert to fire, restores, confirms resolved. Service Links — all URLs and SSH access in one view.

Key point: Stop ALL discovers running VMs dynamically — no hardcoded list. Graceful shutdown first, force destroy after 5s for any stragglers.

GitLab CI/CD Pipeline — 5 Stages

test → build → deploy-staging → integration-test → deploy-prod (manual gate). Dynamic node discovery — pipeline queries k3s API for all node IPs at runtime, no hardcoded lists. Reusable SSH anchor and image prune anchor eliminate duplication. Auto-prunes old images from all nodes after every deploy.

Key point: imagePullPolicy: Never — images must exist on every node before deployment. Artifact expiry set to 1 week to allow manual production deploys days after the pipeline ran.

Observability Stack — Full PLG + 3 Grafana Dashboards

Prometheus scraping all nodes including ci-runner. Loki receiving logs from all k3s pods via Promtail DaemonSet. All dashboards provisioned as ConfigMaps (grafana_dashboard=1 label) — permanent across pod restarts and PVC loss. Three dashboards: All Nodes (CPU/RAM/Disk/Network with node_uname_info hostname join), API Dash (request rate, error rate by HTTP code/verb/resource, top erroring endpoints table, p99 latency, scheduler, work queue, pod restarts), Log Explorer (cluster health row with pod restarts/OOMKills/not-running pods, monitoring namespace errors, custom namespace/pod explorer).

Key point: node_uname_info join trick converts IP addresses to hostnames in all node metrics. Dashboards as code — never lost again.

Alerting Pipeline — Full Lifecycle with Auto-Remediation

Three severity tiers: CRITICAL → #incidents + GitLab issue + auto-remediation, WARNING → #all-homelab-alerts, INFO → #all-homelab-alerts. Dynamic title template shows RESOLVED in green vs CRITICAL in red — visually distinct. Resolved notifications arrive within 60 seconds of metric recovery. Custom webhook.py auto-creates GitLab Issues (INC numbers, severity labels, timeline) on fire and auto-closes on resolve. RUNBOOKS dict in webhook.py drives auto-remediation: NodeMemoryHigh on ci-runner → gitlab-ctl restart via SSH, NodeDiskHigh on any node → crictl rmi --prune. Auto-remediation result (success/fail + output) embedded in the GitLab issue. 6 custom PrometheusRules. Two-tier memory alerting: warning at 80%, critical + auto-fix at 85%.

Key point: Auto-remediation runs before creating the GitLab issue so the issue includes the remediation result. The webhook pod mounts an SSH key secret to reach nodes inside the cluster network.

IaC — Terraform + Ansible

Workers provisioned via libvirt provider with cloud-init. Separate infra.tf for permanent k3s-infra node. Dynamic Ansible inventory regenerated on every scale operation. install-apps.yml targets k8s_cluster group only — ci-runner excluded intentionally. k3s-start.sh fully dynamic — discovers all k3s-worker-* VMs from virsh at runtime, skips missing VMs gracefully.

Key point: infra_enabled = true/false toggle in tfvars controls infra node independently from worker count.

GitLab Memory Tuning

GitLab CE was consuming 8.4GB idle on a 10GB VM — creeping to OOM over time. Tuned via gitlab.rb: puma worker_processes=2 (down from 4), sidekiq concurrency=5 (down from 25), prometheus_monitoring disabled (replaced by kube-prometheus-stack). Result: idle memory dropped from 8.4GB to 3.3GB — a 60% reduction. Nightly cron restarts GitLab at 3am to reclaim any leaked memory before the working day.

Key point: GitLab ships with production-scale defaults. For a single-user homelab, reducing Puma workers and Sidekiq threads is safe and dramatically reduces memory pressure.
04 Issues Encountered & Lessons Learned

Stateful workload on a scalable node (Loki PVC lost)

Loki was initially pinned to k3s-worker-2 — a Terraform-managed node. When the node was downscaled and destroyed, the PVC was deleted with it. Loki lost all logs and failed to restart on the replacement node.

Lesson: Stateful workloads MUST live on permanent nodes. Solution: dedicated k3s-infra node with NoSchedule taint, never part of the scaling pool.

Grafana dashboards lost on pod restart

Grafana deployed without a PVC uses emptyDir. Any pod restart wipes all manually created dashboards. Discovered after a Helm upgrade.

Lesson: Enable grafana.persistence.enabled=true. Better: provision dashboards as ConfigMaps with label grafana_dashboard=1 — version-controlled, survive even PVC loss.

Alertmanager config overwritten on every Helm upgrade

Every helm upgrade overwrote the Alertmanager secret with the default null receiver. Also: a Sprig trunc function in the title template caused all critical alerts to fail — Alertmanager uses standard Go templates, not Sprig.

Lesson: All Alertmanager config must live in fix-values.yaml and always passed with -f flag. Never edit the secret directly.

App pods scheduling on control plane (ErrImageNeverPull)

k3s does not taint the control plane by default. Pods scheduled on k3s-control had no container images. imagePullPolicy: Never means it won't pull — if the image isn't there, it fails.

Lesson: Always taint the control plane in k3s. Pre-sync images to all nodes before scaling.

k3s false-positive alerts (KubeSchedulerDown etc.)

kube-prometheus-stack expects scheduler, controller-manager and kube-proxy as separate pods. k3s embeds all three in the k3s binary — they never appear as pods so the rules fire immediately after install.

Lesson: Disable via Helm defaultRules.disabled. The expr: "false" override approach doesn't work — the original upstream rules still fire alongside it.

Grafana PVC init container permission denied

After enabling Grafana persistence, the init-chown-data container failed with "Permission denied" on /var/lib/grafana — blocking the pod in CrashLoopBackOff.

Lesson: Disable via grafana.initChownData.enabled=false in Helm values. Unnecessary for fresh PVCs.

Alertmanager resolved notifications not arriving

send_resolved: true was set but resolved notifications were invisible — the title template had no resolved indicator so firing and resolved messages looked identical. Also: repeat_interval: 1h was racing against resolve causing timing issues.

Lesson: Use dynamic title template: {{ if eq .Status "resolved" }}RESOLVED{{ else }}CRITICAL{{ end }}. Set group_interval: 30s and resolve_timeout: 30s for fast resolution delivery.

k3s uses SQLite not etcd

The etcd Grafana dashboard was empty. k3s uses SQLite by default for single control-plane setups.

Lesson: k3s SQLite is appropriate for non-HA setups. For production HA: use --cluster-init for embedded etcd, or external datastore.

GitLab memory leak + OOM on ci-runner

GitLab CE leaks memory over time — reached 8.4GB on a 9GB VM causing node-exporter OOM kill, which stopped Prometheus scraping ci-runner, which stopped the NodeMemoryHigh alert from resolving (no metric = no recovery signal).

Lesson: Tune GitLab puma/sidekiq workers down for homelab. Add nightly restart cron. Monitor node-exporter health separately — if it dies, alerts from that node can't resolve.

k3s version mismatch between nodes

k3s-infra provisioned with a newer version than existing nodes because the install script pulled latest.

Lesson: Pin k3s version: INSTALL_K3S_VERSION=v1.33.6+k3s1. In production: consistent version management across all nodes.

GitLab GPG key expiry broke apt on ci-runner

GitLab's package repo signing key expired, cascading to broken apt and missing curl/wget.

Lesson: Third-party apt repositories can silently expire. Refresh keyrings at /etc/apt/keyrings/.
05 Key Interview Talking Points
"I built a homelab that mirrors a real production SRE environment." Everything is automated — infrastructure provisioned by Terraform, nodes configured by Ansible, applications deployed via GitLab CI/CD, observed by a full PLG stack, and alerts routed automatically to Slack and GitLab with auto-remediation.

On the Incident Demo

"Let me show you a live incident. I'll scale the app to zero — within 90 seconds you'll see a CRITICAL alert in Slack, a GitLab issue auto-created with an INC number. When I restore it, the RESOLVED notification arrives in green within 60 seconds and the issue closes automatically. The whole lifecycle is automated."

On Auto-Remediation

"When NodeMemoryHigh fires on ci-runner, the webhook SSHes into the node and restarts GitLab services automatically. The GitLab issue includes the remediation result — success or failure with full output. If it fails, the issue stays open for manual investigation. No human needed for known, recoverable failures."

On Observability

"I have metrics, logs, and alerting integrated. Prometheus scrapes every node. Loki aggregates all pod logs. Three Grafana dashboards: node resources, API server health with error drilldown by verb/resource/code, and a Log Explorer with cluster health panels showing pod restarts, OOMKills, and monitoring namespace errors. All dashboards are provisioned as code — they survive any restart."

On Architecture Decisions

"I learned the hard way that stateful workloads need dedicated permanent nodes. After losing Loki's data during a scale-down, I created a separate infra node with NoSchedule taints — the same pattern used in production OpenShift infra nodes. Similarly, after losing Grafana dashboards to a pod restart, I moved to ConfigMap provisioning — infrastructure as code for observability config."

On k3s vs OpenShift

"k3s gave me hands-on Kubernetes in a resource-constrained environment. Key differences: no etcd by default (SQLite for single-node), no SCCs (just PSA), control plane not tainted by default, scheduler/controller-manager/kube-proxy embedded in the binary. OpenShift adds enterprise security, RBAC depth, operator framework, and HA etcd."

On Resource Management

"Running 4 VMs on a 30GB laptop for an interview required careful tuning. GitLab CE defaults to production-scale settings — I reduced Puma workers from 4 to 2 and Sidekiq concurrency from 25 to 5, cutting idle memory from 8.4GB to 3.3GB. Added a nightly restart cron to reclaim leaked memory. The alerting system itself monitors memory and auto-restarts GitLab if it breaches 85%."
