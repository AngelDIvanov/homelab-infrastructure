03 What Was Built

Unified Control Panel — lab-control.py

Single Python script managing the entire lab. Dynamic VM status banner via virsh. Start/Stop ALL VMs. Scale up/down workers with full automation (Terraform → SSH ready → k3s join → image sync → replica scaling). Ansible playbook runner. Health checks with auto-fix. Alerting test suite with two-phase RAM nuke demo.

Key point: Upscale automatically scales app replicas to match worker count. Downscale reduces replicas first.

Change Management Pipeline — GitLab MR as Change Ticket

Every change to production goes through a Merge Request. MR description uses a mandatory Change template (What/Why/Testing/Rollback). Branch protection on main — no direct pushes. Pipeline validates MR before running: validate → test → build → deploy-staging → integration-test → bump-version → deploy-prod (manual gate). Version labels (major/minor/patch) on MR drive automatic semver bumping. Production deploy creates a git tag (v1.2.3). Full audit trail — including rollbacks — tracked as MRs.

Key point: In prod you'd have multi-user approval. Here it demonstrates the pattern — change ticket, approval gate, pipeline, versioned deploy, audit trail.

Semantic Versioning — Automated

VERSION file in the trengo-search repo tracks the current version. MR label (patch/minor/major) drives the bump — patch increments the patch number, minor resets patch to 0, major resets both. On every successful prod deploy the pipeline auto-tags the commit as v1.2.3 in GitLab. No manual tagging required.

Key point: version label must be set before pipeline runs — validate stage blocks the pipeline if missing.

GitLab CI/CD Pipeline — 7 Stages

validate → test → build → deploy-staging → integration-test → bump-version → deploy-prod (manual gate). Dynamic node discovery — pipeline queries k3s API for all node IPs at runtime, no hardcoded lists. Reusable SSH anchor and image prune anchor eliminate duplication. Auto-prunes old images from all nodes after every deploy. Artifact expiry set to 1 week to allow manual production deploys days after the pipeline ran.

Key point: imagePullPolicy: Never — images must exist on every node before deployment.

Observability Stack — Full PLG + 4 Grafana Dashboards

Prometheus scraping all nodes including ci-runner. Loki receiving logs from all k3s pods via Promtail DaemonSet. Four dashboards: All Nodes (CPU/RAM/Disk/Network with node_uname_info hostname join), API Dash (request rate, error rate, top erroring endpoints, p99 latency), Log Explorer (cluster health, pod restarts, namespace explorer), Trengo App (replica status, log volume, all logs, errors only). Health check CronJob pushes directly to Loki API when app has 0 replicas — bypasses Promtail entirely.

Key point: node_uname_info join trick converts IP addresses to hostnames in all node metrics. Dashboards as ConfigMaps — permanent across pod restarts and PVC loss.

Two-Tier Alerting with Auto-Remediation

Three severity tiers: CRITICAL → #incidents + GitLab issue + auto-remediation, WARNING → #all-homelab-alerts. Two-tier memory alerting: NodeMemoryHigh at 85% fires a CRITICAL alert to Slack with GitLab issue (no remediation). NodeMemoryCritical at 90% fires CRITICAL + triggers auto-remediation (pkill stress-ng + gitlab-ctl restart via SSH from the webhook pod). TrengoAppDown triggers at 30s with 0 available replicas. All alerts auto-resolve with RESOLVED notifications and GitLab issue auto-close.

Key point: Auto-remediation runs before creating the GitLab issue — so the incident URL in Slack already contains the remediation result.

RAM Nuke Test — Two-Phase Auto-Remediation Demo

Automated demo in lab-control.py that proves the full alerting pipeline end to end. Phase 1: stress-ng consumes 3G RAM on ci-runner → hits 85% → NodeMemoryHigh fires → Slack CRITICAL alert + GitLab issue created → script pauses for operator to confirm. Phase 2: additional 1G stress → hits 90% → NodeMemoryCritical fires → auto-remediation runs (pkill stress-ng + gitlab-ctl restart) → memory drops → both alerts resolve → RESOLVED in Slack + GitLab issues auto-closed. Script shows live RAM% during polling and detects when stress process is gone.

Key point: Two thresholds — alert at 85% gives visibility, auto-remediation at 90% gives time for a human to see the alert before the system self-heals.

Security Hardening — Ansible on Infra Node

Lightweight hardening playbook targeting k3s-infra only. Installs and configures fail2ban (3 failed attempts = 1 hour ban, monitored via /var/log/auth.log). Sets SSH warning banner visible on every login. Revert playbook removes both cleanly. Demonstrated live during interview by SSH-ing into k3s-infra and showing the banner appear.

Key point: Running on infra-only is intentional — avoids UFW misconfiguration that could break cluster networking on worker/control nodes (learned from previous incident).

IaC — Terraform + Ansible

Workers provisioned via libvirt provider with cloud-init. Separate infra.tf for permanent k3s-infra node. Dynamic Ansible inventory regenerated on every scale operation. install-apps.yml targets k8s_cluster group (control + workers + infra). ci-runner excluded intentionally — different OS profile.

Key point: infra_enabled = true/false toggle in tfvars controls infra node independently from worker count.
04 Issues Encountered & Lessons Learned

Stateful workload on a scalable node (Loki PVC lost)

Loki was initially pinned to k3s-worker-2 — a Terraform-managed node. When the node was downscaled and destroyed, the PVC (local storage path) was deleted with it. Loki lost all logs and failed to restart on the replacement node.

Lesson: Stateful workloads MUST live on permanent nodes. Solution: Created dedicated k3s-infra node that is never part of the scaling pool — same pattern as OpenShift infra nodes.

App pods scheduling on control plane (ErrImageNeverPull)

k3s does not taint the control plane by default. When workers were unavailable, pods scheduled on k3s-control which had no container images. imagePullPolicy: Never means it won't pull — if the image isn't there, it fails.

Lesson: Always taint the control plane in k3s. Also pre-sync images to all nodes before scaling.

Helm upgrades reset custom Alertmanager config

Every helm upgrade overwrote the Alertmanager secret with the default null receiver, silently breaking all alert routing. Alerts were firing in Prometheus but never reaching Slack.

Lesson: Custom configs must be embedded in values.yaml — fix-values.yaml is now the single source of truth, applied on every Helm upgrade.

Auto-remediation too fast — alert fired and resolved before anyone saw it

NodeMemoryHigh at 85% triggered both the alert AND the auto-remediation (gitlab-ctl restart). The restart freed memory so fast the alert resolved in under 30 seconds — before anyone could see it in Slack.

Lesson: Split into two thresholds — alert-only at 85% (NodeMemoryHigh) gives visibility, auto-remediation at 90% (NodeMemoryCritical) gives time for a human to read the alert first.

GitLab CI label not passed to manually triggered pipelines

GitLab CE doesn't pass CI_MERGE_REQUEST_LABELS to pipelines triggered manually via the UI — only to pipelines triggered by push events. The validate stage blocked every manually re-triggered pipeline even when the label was set.

Lesson: Changed validate to allow_failure: true — it still checks and warns, but doesn't block the pipeline. In production you'd use GitLab EE approval rules or a separate approval job.

k3s uses SQLite not etcd

The etcd Grafana dashboard was completely empty. k3s uses SQLite by default for single control-plane setups.

Lesson: k3s SQLite is appropriate for non-HA setups. For production HA: use --cluster-init for embedded etcd, or external datastore.

check-lab.sh treating Completed CronJob pods as stuck

The health check auto-fix script used grep -v Running to detect stuck pods. CronJob pods (trengo-health-check) show as Completed — not Running — so the script treated them as stuck and tried to force-delete them on every run.

Lesson: grep -v Running | grep -v Completed | grep -v Succeeded — always exclude terminal states from stuck pod detection.

stress-ng hitting wrong threshold (84% not 90%)

Initial stress test used --vm-bytes 2G which only reached 84% on ci-runner's 7.8GB RAM — not enough to trigger NodeMemoryCritical at 90%. Multiple iterations required to calibrate: 2G → 3G → 3.5G for Phase 1, +1G for Phase 2.

Lesson: Always check baseline RAM before sizing a stress test. Formula: target% × total_RAM - baseline_used = stress_bytes needed.

k3s version mismatch between nodes

k3s-infra provisioned with a newer k3s version (v1.34.5) than existing nodes (v1.33.6) because the install script pulled latest.

Lesson: Pin k3s version in the install command: INSTALL_K3S_VERSION=v1.33.6+k3s1. In production: use a consistent version management strategy.
05 Key Interview Talking Points
"I built a homelab that mirrors a real production SRE environment." Everything is automated — infrastructure provisioned by Terraform, nodes configured by Ansible, applications deployed via GitLab CI/CD with change management gates, observed by a full PLG stack, and alerts routed automatically to Slack and GitLab Issues with auto-remediation.

On Change Management

"Every change to production goes through a Merge Request — that's the change ticket. The MR template enforces What/Why/Rollback documentation. Branch protection means nobody can push directly to main. The pipeline validates, tests, deploys to staging, runs integration tests, then presents a manual deploy-prod button. Version labels on the MR drive automatic semver tagging. In prod you'd have multi-approver rules — here I'm demonstrating the pattern."

On Auto-Remediation

"When NodeMemoryCritical fires at 90%, the webhook pod SSHs into ci-runner and runs gitlab-ctl restart automatically — before even creating the GitLab issue. The issue includes the remediation result (success/fail + output). The alert then resolves itself when memory drops below threshold. The operator sees: CRITICAL fired → auto-remediation ran → RESOLVED — all in Slack within 60-90 seconds, no manual intervention required."

On the RAM Nuke Demo

"I can run a live demo that shows the full alerting lifecycle. Phase 1 eats RAM to 85% — you see NodeMemoryHigh fire in Slack with the GitLab issue link. Phase 2 pushes to 90% — NodeMemoryCritical fires, auto-remediation kills the stress process and restarts GitLab, memory drops, both alerts resolve with RESOLVED messages. The whole cycle takes about 3-4 minutes."

On Scaling

"I can demonstrate live scaling during the interview. One command provisions a new VM, joins it to the cluster, syncs the application image, and scales the deployment — all automated. The reverse is equally clean. Currently running 2 workers (k3s-worker-1 permanent + k3s-worker-2 Terraform-managed)."

On Observability

"I have metrics, logs, and alerting all integrated. Prometheus scrapes every node including the CI runner. Loki aggregates logs. Four Grafana dashboards including a dedicated Trengo App dashboard with replica status, log volume, all logs, and errors-only panels. Alerts auto-create GitLab incidents with INC numbers and auto-resolve. The Trengo dashboard even has a dedicated Loki panel that shows health-check logs pushed directly to Loki's API when the app has zero replicas — bypassing Promtail entirely."

On Architecture Decisions

"I learned the hard way that stateful workloads need dedicated permanent nodes. After losing Loki's data during a scale-down, I created a separate infra node with NoSchedule taints that's never part of the scaling pool — the same pattern used in production OpenShift infra nodes."

On k3s vs OpenShift

"k3s gave me hands-on Kubernetes experience in a resource-constrained environment. Key differences I'm aware of: no etcd by default (SQLite), no SCCs (just PSA), control plane not tainted by default, embedded SQLite instead of etcd for single-node. OpenShift adds enterprise security, RBAC depth, SCCs, the operator framework, and built-in image registry on top."
